This guide will take you from zero to your first successful API call in just a few minutes. By the end, you'll have a working client, fetched live data from the Voltus platform, and created your first resource. No prior experience with the SDK is required.

## Introduction

The Voltus SDK is designed to feel natural to JavaScript and TypeScript developers. If you've ever used a library like Stripe's SDK or the AWS SDK, the patterns will be immediately familiar. You create a client, call methods on it, and get back typed responses. Every method is async, every error is structured, and every response follows a consistent shape.

### How It Works

Under the hood, the SDK wraps the Voltus REST API in a clean, type-safe interface. When you call `client.resources.list()`, the SDK constructs an HTTP request, sends it to the API, parses the response, and returns a fully typed JavaScript object. It also handles authentication headers, retry logic, rate limiting, and error parsing automatically.

### What You'll Build

By the end of this tutorial, you'll have a working script that connects to the Voltus API, retrieves a list of resources, creates a new resource, updates it, and then cleans up by deleting it. These four operations — list, create, update, delete — form the foundation of every integration.

## Step 1 — Initialize the Client

The first thing every application needs is a configured client instance. This is the single object through which all your API interactions will flow.

### Create the Client Module

Create a new file called `client.ts` in your project. Import the `VoltusClient` class from the SDK and create a new instance, passing your API key from the environment variable. Export this instance so other files can import and use it without creating their own clients.

### Configuration Options

The client accepts several optional settings beyond the API key. You can set the `environment` to `sandbox` for testing, adjust the `timeout` for slow networks, or change the `retries` count for flaky connections. For now, the defaults are perfectly fine — you can always adjust them later as your needs evolve.

## Step 2 — Fetch Resources

With the client ready, you can start reading data from the platform. List operations are the most common starting point for any integration.

### Making the Request

Call `client.resources.list()` to fetch the first page of resources. The response object contains two properties — `data`, which is an array of resource objects, and `pagination`, which tells you the total count, current page size, and whether more pages are available. Each resource in the array has an `id`, `name`, `description`, `status`, `tags`, and timestamp fields.

### Understanding the Response

The pagination object follows a consistent pattern across every list endpoint in the SDK. The `total` field tells you how many records exist in total, `limit` is how many were returned in this page, `offset` is where this page started, and `hasMore` indicates whether there are additional pages to fetch. This consistent structure means you only need to learn the pattern once.

## Step 3 — Create a Resource

Reading data is useful, but most integrations also need to write data back to the platform. Creating a resource demonstrates the SDK's mutation capabilities.

### Building the Payload

Call `client.resources.create()` with an object containing the fields for your new resource. The only required field is `name` — everything else is optional. You can include a `description`, an array of `tags` for organization, and a `metadata` object for storing arbitrary key-value pairs that are specific to your application.

### Handling the Response

The create method returns the newly created resource with all of its server-generated fields populated — including the unique `id`, the `status` which defaults to `draft`, and the `createdAt` and `updatedAt` timestamps. You can use the returned `id` immediately for subsequent operations like updates or deletions.

## Step 4 — Update and Delete

The final operations complete the CRUD lifecycle. Updating lets you modify existing resources, and deleting removes them permanently.

### Updating Fields

Call `client.resources.update()` with the resource ID and an object containing only the fields you want to change. You don't need to send the entire resource — the SDK uses PATCH semantics, so only the specified fields are modified and everything else remains unchanged. The method returns the full updated resource so you can verify the changes.

### Deleting a Resource

Call `client.resources.delete()` with just the resource ID. The method returns void on success and throws a `NotFoundError` if the resource doesn't exist. Deletion is permanent and cannot be undone, so make sure you're targeting the correct resource before calling this method in production code.
